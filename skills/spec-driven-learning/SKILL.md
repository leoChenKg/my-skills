# Spec 驱动学习工作流

## 项目结构

```
openspec/
├── project.md               # 全局学习规划（每次操作前必须先读）
└── changes/
    ├── add-phase-N-xxx/     # 当前进行中的学习阶段
    │   ├── proposal.md      # 本阶段目标与动机
    │   ├── tasks.md         # 学习任务清单
    │   └── specs/
    │       └── <topic>/spec.md
    └── archive/             # 已完成阶段
src/
└── phase-N/                 # 产出：练习代码 + 学习笔记(.md)
```

---

## 三阶段工作流

### 阶段一：规划（Propose）

**触发词：**「开始 Phase N」「创建提案」「规划」「我要学 xxx」

**步骤：**
1. 读 `openspec/project.md`，确认当前阶段目标与学习者背景
2. 在 `openspec/changes/add-phase-N-xxx/` 下创建：
   - `proposal.md`：Why（为什么学）+ What（学什么）+ 与已有知识的关联
   - `tasks.md`：有序可勾选任务列表（见格式规范）
   - `specs/<topic>/spec.md`：每个知识模块的详细规格
3. **规划阶段不写任何代码**，只产出规划文档
4. 列出提案摘要，**等待用户审核确认**后才进入实施阶段

### 阶段二：实施（Apply）

**触发词：**「开始执行」「apply」「开始学习」「继续」

**步骤：**
1. 读 `proposal.md` 确认目标，读 `tasks.md` 获取任务列表
2. **按顺序**逐条执行，每完成一项立即标记 `[x]`
3. 代码产出写入 `src/phase-N/`（文件名对应任务编号，如 `01-variables.py`）
4. 笔记写入 `src/phase-N/xx-topic.md`，包含：
   - 与已有知识的差异总结（重点）
   - 踩坑记录及解决方式
   - 可运行的代码示例
5. 全部完成后列出本阶段收获摘要，**等待用户验收**

### 阶段三：归档（Archive）

**触发词：**「归档」「archive」「完成本阶段」

**步骤：**
1. 将 `openspec/changes/add-phase-N-xxx/` 整个移入 `archive/YYYY-MM-DD-add-phase-N-xxx/`
2. 在 `project.md` 中将该阶段标记为 `✅ 已完成`
3. 确认 `src/phase-N/` 下有完整的代码和笔记产出
4. 告知用户下一阶段可以开始

## 文件格式规范

### tasks.md 格式

```markdown
## 1. <模块名>（<topic-id>）

> ⭐⭐⭐ | 与已知知识的关键差异：[一句话说明]

- [ ] 1.1 [具体学习任务，包含明确的验收标准]
- [ ] 1.2 [...]
- [ ] 1.3 练习：[明确的代码练习题目]
- [ ] 1.4 撰写学习笔记 `src/phase-N/01-topic.md`

⚠️ PITFALL：[本模块最常见的错误和误解]

## 2. <下一个模块>
...

## N. 阶段验收
- [ ] N.1 回顾本阶段所有 ⭐⭐⭐ 知识点，确认有代码实践
- [ ] N.2 整理学习心得到 `design.md`
```

### proposal.md 格式

```markdown
# Change: 添加 Phase N — [阶段名称]

## Why
[1-2 句话：为什么要学这个，学完能做什么]

## What Changes
- 新增 `<topic>` spec：[知识点说明]
- 新增 `<topic>` spec：[...]

## Impact
- 关联知识：[与已掌握知识的关系]
- 产出位置：`src/phase-N/`
- 预计时长：[X 天 / X 小时]
```

### spec.md 格式

```markdown
# Spec: <知识点名称>

## 定义
[简洁的概念说明]

## 与已知概念对比
| 已知（JS/其他） | 当前（Python/新技术） |
|---|---|
| [对应概念] | [新概念及差异] |

## 使用场景
- 场景一：[什么时候用]
- 场景二：[...]

## 代码示例
[可运行的最小示例]

## ⚠️ 常见坑点
[易错点说明]
```

## 核心原则

| 原则 | 说明 |
|------|------|
| **差异优先** | 重点讲与已有知识的不同之处，相似概念快速略过 |
| **产出验证** | 每个知识点必须有可运行代码或可验证笔记 |
| **坑点前置** | 每个模块必须标注 ⚠️ PITFALL |
| **优先级分层** | ⭐⭐⭐ 必须精通 / ⭐⭐ 需要掌握 / ⭐ 了解即可 |
| **阶段门控** | 规划须审核后才执行，执行须验收后才归档 |
| **单阶段聚焦** | 同时只有一个 change 进行中 |

## 快速触发词对照表

| 用户说 | AI 执行动作 |
|--------|------------|
| 「开始 Phase N」/ 「我要学 xxx」 | 进入规划阶段，生成 proposal + tasks + specs |
| 「开始执行」/ 「apply」 | 进入实施阶段，按 tasks.md 逐条完成 |
| 「下一个」/ 「继续」 | 执行 tasks.md 中下一条未完成的任务 |
| 「查看进度」 | 读取当前 change 的 tasks.md，展示完成情况 |
| 「归档」/ 「archive」 | 归档当前阶段，更新 project.md |
| 「初始化项目」 | 生成 project.md 全局规划框架 |
