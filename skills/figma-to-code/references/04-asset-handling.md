# 04 · 资源处理（用户提前准备 + 语义命名 + 扫描引用）

目标：资源由**用户提前下载、语义化命名**放入项目，代码生成时由 agent **扫描资源目录按语义名匹配引用**现存资源；缺失则**就地按单元提示补齐(jit)**。**不维护资源清单、不在结构分析阶段处理资源**（约束 B3/B7）。

> 流程位置：资源准备是**用户侧前置**（步骤 2 之后、步骤 3 实现前）；agent 的引用动作发生在步骤 3 的 **3b 代码生成**。

## 通用铁律（不可违背）

B1~B7 全文以 SKILL.md「全局约束」B 节为准；本文是其落地（背景填充图见下「背景填充图」，用户准备与扫描引用见下两节）。补充提醒：

- Figma asset URL **约 7 天失效**，故 B2 要求项目代码只引本地路径（预览的原始 `.tsx` 例外，见 [05-visual-verification.md](05-visual-verification.md)）。
- 禁占位图 / 占位色块代替真实资源；禁为单个图标新增整套图标依赖包；同一资源按内容去重。

## 资源不进结构分析阶段

结构分析（步骤 1）**只产出模块拆分 / 依赖 / 复用 / 批次，不探索资源、不列资源**。资源在实现期才处理，避免提前生成一份很快失效、维护成本高的资源清单。

## 两条分支（步骤 0/2 选定，默认 B）

资源"怎么进目录"有两条可选路径；**下游 3b 引用方式两分支完全一致**（都是扫描资源目录按语义名引用现存资源、缺失 jit），分支只影响"谁、怎么把资源放进目录"。未指定默认走 B（不破坏现有习惯）。

### 分支 B · 用户自备（默认，现状不变）

由用户下载 + 语义命名 + 放目录 + 告知路径，agent 只扫描引用、缺失 jit。`B3` 在此分支即"agent 不下载"。详见下「用户准备」。

### 分支 A · agent 协助导出

agent 只做**确定性导出**、把**判断性决策交用户裁决**：

1. **识别确定的图节点**：从步骤 2.5/3a 确认过的 reference `.tsx` 找出确定是 `<img>` 的节点（CSS 能实现的先按 `B1` 剥离，不导出）。
2. **导出**：对这些节点用 `get_screenshot(nodeId, contentsOnly:true, maxDimension=节点长边)`（契合 F3 清晰度）导出 PNG，按语义名落到步骤 0 探测到的资源目录。
3. **产出 `asset-manifest`（三类标注）**：
   - ① **已导出**：节点 → 文件名。
   - ② **已剥离为 CSS**（按 `B1`，渐变/圆角/阴影等，不导出）。
   - ③ **待用户确认**：整体导出 vs 拆碎、透明处理、`@2x`/分辨率、背景填充图（`B6`）判定——这些**判断性**决策 agent 不擅自定。
4. **用户只裁决第 ③ 类**，确认/修正后继续。`B3` 在此分支放宽为"agent 可导出确定 `<img>` 节点，但切图边界/透明/分辨率以用户对 manifest 的确认为准"；`B1/B2/B4/B6/B7` 不变。

> 为什么不全自动：切图边界、透明、分辨率是**判断**，agent 易错（见用户反馈）。分支 A 只替你做"确定的体力活"，判断仍归你。

## 用户准备（实现前置 · 分支 B）

由用户完成，agent 只接收"资源目录已就绪"的信息：

1. **自行下载**所需资源：分辨率、`@2x`、透明处理等由用户按设计决定（agent 不下载、不决定分辨率）。
2. **语义化命名**：文件名要让 AI 一看即知**用途与位置**，例如：
   - `core-class-title-bg.png`（核心课堂标题背景填充图）
   - `nav-tab-home-icon.svg`（导航首页 tab 图标）
   - `banner-hero@2x.png`（顶部 Banner 主视觉）
3. **放入项目对应目录**：按项目既有资源目录约定存放（步骤 0 探测到的）。
4. **把资源目录路径告诉 agent**：生成代码前提供，供 3b 扫描。

## 3b 扫描引用（agent 侧）

代码生成时，对每个待建单元需要的资源：

1. 从步骤 2.5/3a 确认过的 reference `.tsx` / `get_design_context` 识别该单元用到哪些图片资源（CSS 能实现的先按 B1 剥离，不算资源）。
2. **扫描用户提供的资源目录**，按语义名 + 设计用途匹配现存文件。
3. **命中** → 引用该本地路径（B2：不写在线链接、不 base64）。
4. **缺失** → **就地停下、按单元提示用户**：「单元 `<nodeId>` 需要资源 `X`（用途：…），请按语义命名放到 `<约定路径>`」；用户补齐后继续，**不回退到全局清单**。

## 切图边界与透明处理

- **整体导出判定**：可整体导出成一张图的区块就整体导出，别拆碎；但能 CSS 表达的部分先剥离（B1）。这点提示用户在准备资源时一并参考。
- **透明图**：透明位图建议用户在导出时用 `get_screenshot(contentsOnly:true)` 思路隔离目标节点，**避免把底层背景混进图里**；命名上可加 `-transparent` 等语义后缀。
- **分辨率**：由用户按需下载；agent 用于**分析/判定**的截图本身须清晰（约束 F3，见 [05-visual-verification.md](05-visual-verification.md)），与用户下载的资源分辨率是两回事。
- **SVG 用 `<img>` 引入时的 `var(--fill-N)`**：Figma 导出偶尔在 SVG 里写 `fill="var(--fill-0, #xxx)"`。`<img>` 标签里的 SVG 是**独立文档**，外部 CSS 变量**不会**透进去——实际生效的是兜底色（逗号后的值）。所以：① 预览/交付若发现颜色不对，先确认兜底色是否正确；② 兜底色不对时改用**内联 `<svg>`**（变量可继承）或直接把目标色写进资源。本方案当前未触发此坑（导出的渐变 SVG 兜底色正确）。

## 成品图禁反推内部（约束 B6，覆盖一切用户预备成品图）

**核心铁律**：用户预备的成品图（背景填充图 / 图标 / 徽章 / 装饰图 / 照片等）**自带取景、留白与视觉效果**，与 Figma 内部矢量包围盒并不一致。agent 只负责**把它放进容器并施加外层定位**，**不重算其内部布局**。**成品图的尺寸/位置只能来自设计稿（设计节点的摆放框），不能来自成品图自身。**

- **展示尺寸的唯一来源 = 设计节点摆放框**：成品图的展示宽高 = 该资源在设计里所属节点的**外层摆放框**（`get_metadata` / 原始 `.tsx` 容器的 `w`/`h`，如 `章-大 = 84×72`、位置 `left:255 top:0`、容器 `overflow:hidden`）。图按 `width:100%` 铺满这个框 + 仅施加外层越界偏移；越界部分由容器 `overflow:hidden` 裁掉。
- **成品图优先规则（视觉效果）**：只要引用的是用户已备成品图片，就**禁止继承 Figma 内部图层的 `opacity` / `filter` / `mask` / `transform` 等效果**；这些效果默认已经烘焙在成品图里。只有用户明确说明该资源是不含这些视觉效果的**裸素材**时，agent 才能把对应效果按设计稿补到外层容器，并在卡点说明。
- **禁止的错误来源**：
  - ① 把 Figma 内部矢量 wrapper 的 `transform` / `inset` / `bbox` / `wrapper` 数值转写到成品图上来定位/定尺寸。
    - 实战教训：处理「章-大」徽章时，把内部矢量 wrapper 的变换值（`w≈93.338 / left≈-5.99 / top≈-16.61`）转写到了成品图 `stamp.png` 上，导致摆放错误。
  - ② **用切图/位图文件自身的固有/导出像素当尺寸**（`sips` / `naturalWidth` / `pixelWidth` 等读出的文件像素）——文件像素只是导出产物（含导出取景、透明留白、`@2x` 倍率），**与设计展示尺寸无关**，禁作任何样式值来源。
    - 实战教训：把 `stamp.png` 的文件像素 `112×112` 当成展示尺寸写进 CSS，还把设计真实摆放框 `84×72` 误判成"只是裁剪窗口"丢弃——正好把唯一权威来源舍掉、用了禁用来源。正确：容器 = 设计摆放框 `84×72`，图 `width:100%` 铺满 + 外层越界偏移。
  - ③ 把 Figma 内部图层的 `opacity` / `filter` / `mask` / `transform` 套到用户成品图上，导致成品图被二次处理。
    - 实战教训：处理高光时刻右上角 `highlight-decorate.png` 时，把 `.tsx` 内部层的 `opacity-50` 继续写到用户已备成品图上，导致装饰图被错误变淡。正确：成品图按设计摆放框原样展示；若需要半透明，应由用户在成品图里烘焙或明确说明这是裸素材。
- **正向规则（两类成品图）**：
  - **背景填充图**（整块区域背景）：整图作**容器背景铺满**——`background-size:100% 100%; background-position:center;`，宽随内容自适应。
  - **其余成品图**（图标/徽章/装饰图）：容器 = 设计摆放框尺寸，图按**容器尺寸铺满（`width:100%` / 原始比例）+ 仅施加外层摆放偏移**（如越界 `top`/`left`），越界由 `overflow:hidden` 裁切。

### 背景填充图判定线索

- 该图节点位于父容器**最底层 z 序**、其包围盒 **≈ 父容器框**、有**文字/内容叠在其上** → 按"背景填充图"铺满处理。
- **约定**：用户提供的成品图都已计算好内部元素位置与留白，**直接使用即可**，agent 不再获取其内部元素另行计算。

## 分类决策树

```text
拿到一个 Figma 资源需求：
  能用 CSS 实现?
    → 能 → 用 CSS（渐变/圆角/阴影/描边/纯色/简单形状），不用图
    → 不能 ↓
  是整块区域的背景图?（最底层 z 序 / 框≈父容器 / 内容叠其上）
    → 是 → 背景填充图：整图作容器 background 铺满（background-size:100% 100%），不反推内部
    → 否 ↓
  可复用单色图标?
    → 先查项目图标库（复用优先）；单色用 currentColor 跟随文字色
      （多色图标 / 品牌 logo 保留原始 fill，不套 currentColor）
    → 缺失 → 提示用户语义命名放入项目（jit）
  位图 / 照片 / 复杂矢量 / 徽章 / 装饰图?
    → 扫描资源目录命中 → 引用本地路径，按"成品图"处理：
        容器=设计摆放框尺寸(get_metadata w/h)；图 width:100% 铺满 + 仅外层偏移
        禁继承内部 opacity/filter/mask/transform；禁反推内部矢量 bbox/inset/transform；禁用文件固有像素(sips/naturalWidth)当尺寸（B6）
    → 缺失 → 分支B: 提示用户准备(jit) / 分支A: agent get_screenshot 导出 + manifest 标待确认
```

> **贯穿全树（B6）**：任何用户预备成品图都自带取景/留白/视觉效果，铺满容器 + 仅外层定位，**禁把 Figma 内部矢量数值或 `opacity`/`filter`/`mask`/`transform` 等内部效果转写到成品图**。
> 「缺失」分支：分支 B 提示用户补；分支 A 由 agent 导出确定项、不确定项进 manifest 待用户确认（见上「两条分支」）。

## 要点

- 复用优先级永远高于新增，尤其图标——先扫项目既有资源再提示用户补。
- 单色 vs 多色的 `currentColor` 别搞反，否则品牌色被文字色覆盖。
- 缺失资源是**按单元即时提示**，不攒成一张全局清单——体验更顺，也不产生很快失效的链接表。
